# 04-联调与验收

## 目标
- 在前端工程、聊天主链路和文档生成结果链路完成后，进行系统化联调与验收收口。
- 让当前 MVP 具备"可演示、可验证、问题清单明确"的交付状态。

## 当前状态
- 后端基础接口已存在，并可通过 `/docs` 查看契约。
- 前端主链路尚未真正实现，本分计划当前属于验收准备文件。
- 已知后端限制：
  - 聊天回复为 Mock
  - 文档 `content` 字段始终为空
  - 文档导出接口未完成
  - 部分 README 描述与实际落地不一致
- 已知 Agent 限制：
  - Agent 模块尚未开发，所有智能分析能力不可用
  - 案件要素提取/编辑、法律检索、案例搜索均无接口

## 前置交付确认
- [x] 首页可创建会话并跳转
- [x] 聊天页可发消息并看到回复
- [x] 聊天页有"生成诉状"按钮
- [x] 生成后跳转到结果页
- [x] 结果页展示文档信息和状态
- [x] 下载按钮有占位提示

## 范围

### 纳入
- 对照 `/docs` 核对前端实际请求与响应使用。
- 检查主链路中的空状态、加载态、错误态。
- 检查页面切换、状态同步和基础可用性。
- 形成问题清单与验收结论。

### 不纳入
- 后端业务逻辑重写。
- Agent 能力补全。
- 性能压测和覆盖率建设。
- 复杂自动化测试。

## 联调对象
- 会话创建接口
- 获取会话接口
- 发送消息接口
- 消息历史接口
- 文档生成接口
- 文档详情接口
- 导出接口
- 健康检查接口

## 核对原则
- 一切以前端真实请求和 `/docs` 为准。
- README、注释、口头预期和真实接口不一致时，以真实接口为准，差异记入问题清单。
- 不在联调阶段臆造后端能力。

## 任务拆解

### 1. 接口核对
- 核对前端请求路径、方法、请求体字段。
- 核对响应字段是否与前端类型定义一致。
- 核对错误码和错误文案是否被正确处理。

### 2. 主链路手测
- 创建会话 → 进入聊天页 → 发送消息 → 看到回复 → 生成诉状 → 查看结果 → 点击下载。

### 3. 状态与异常检查
- 无会话时页面表现。
- 无消息时空状态。
- 请求中加载提示。
- 请求失败后恢复。
- 会话/文档不存在的提示。

### 4. 可演示性检查
- 布局完整，文案中文且不混乱。
- 无脚手架残留。
- 无白屏、未处理异常、按钮失效无反馈。
- UI 整体专业、精致。

### 5. 问题清单整理
区分：
- 必须修复的问题
- 可接受但需记录的限制（如 Mock 回复、content 为空、导出未实现）
- 需后端/Agent 后续配合的问题

每个问题项记录：现象、复现路径、影响范围、临时处理方式。

## 验收清单
- 能从首页创建会话并进入聊天页。
- 能发送消息并看到助手 Mock 回复。
- 能触发文档生成并看到结果页。
- 下载按钮行为与后端能力一致，不误导。
- 正文区域展示占位提示，不是空白。
- 页面具备加载、空状态、错误提示。
- 已形成问题清单。
- UI 达到可演示标准。

## 输出要求
- 在 `FRONTEND_WORKFLOW.md` 中补联调/验收记录。
- 在相关分计划中更新完成状态。
- 差异记录在问题清单中。

## 默认决策
- 以手测为主，不要求先上自动化测试。
- 先保证主链路稳定，再考虑体验优化。
- "后端/Agent 未实现的能力"不算前端 Bug，但必须在验收里明确记录。
