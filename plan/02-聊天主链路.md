# 02-聊天主链路

## 目标
- 基于现有后端接口，完成前端聊天主链路的第一版可用实现。
- 用户能够从进入页面开始，完成案件类型选择、会话创建、消息发送、历史查看和助手回复展示。
- 本阶段以"链路跑通 + UI 像模像样"为优先，不等待真实 Agent 接入。

## 当前状态
- 后端已提供聊天相关接口：
  - `POST /api/v1/sessions`
  - `GET /api/v1/sessions/{session_id}`
  - `POST /api/v1/sessions/{session_id}/messages`
  - `GET /api/v1/sessions/{session_id}/messages`
- 当前后端返回特点如下：
  - 创建会话接口返回会话对象。
  - 发送消息接口只返回刚写入的用户消息，不直接返回助手回复。
  - 助手回复会在后端内部追加写入消息表，需要重新请求历史消息才能拿到。
  - 历史消息按时间升序返回。
  - 当前助手回复固定为 Mock 文本（`"感谢您的输入。请继续补充更多信息。"`）。
- Agent 模块未开发，不会有真实的智能回复、意图识别或要素提取。
- 当前前端工程尚未实现，因此本分计划依赖 `plan/01-前端工程初始化.md` 先完成。

## 范围

### 本阶段纳入
- 首页中的案件类型选择和开始咨询入口。
- 创建会话。
- 进入会话后的聊天页面布局。
- 展示历史消息列表。
- 发送消息并同步助手回复。
- 加载态、空状态、失败态和会话不存在等基础错误处理。

### 本阶段不纳入
- 会话列表页。
- 多会话切换。

- 文档生成结果展示细节（属于阶段 3）。

## 接口 / 依赖

### 1. 创建会话
请求：

```json
{
  "case_type": "wage_arrears | labor_contract | work_injury"
}
```

补充说明：
- 后端接口当前仍接受可选 `description` 字段。
- 但当前前端方案采用方案 A：首页只选择案件类型，不在首页额外收集简要描述。
- 用户进入聊天页后，直接在消息输入框中开始描述具体案情。

响应关键字段：
- `id`
- `case_type`
- `status`
- `description`
- `created_at`
- `updated_at`

注意：后端 `case_type` 是 `str` 类型，未做枚举校验。前端应在 `types` 中自定义枚举约束。

### 2. 发送消息
请求：

```json
{
  "content": "用户输入内容"
}
```

响应关键字段：
- `id`
- `session_id`
- `role`（固定为 `"user"`）
- `content`
- `timestamp`

注意：
- 此接口返回的是用户消息本身。
- 助手回复不会在此响应中直接返回。
- 前端必须在此接口成功后立即调用消息历史接口获取助手回复。

### 3. 获取消息历史
请求方式：
- `GET /api/v1/sessions/{session_id}/messages`
- 支持 `limit`、`offset` 查询参数

响应关键字段：
- `session_id`
- `messages`（消息数组）
- `total`

消息结构字段：
- `id`
- `session_id`
- `role`（`"user"` 或 `"assistant"`）
- `content`
- `timestamp`

### API 服务层使用约定
- 所有接口调用通过 `services/chatService.ts` 封装，不在页面组件中直接写 fetch。
- 调用 `services/request.ts` 中的基础方法（`get`、`post`）。
- 错误通过 `message.error()` 展示给用户。

## 页面与交互决策

### 1. 首页（路由 `/`）
- 展示项目名称和简要说明。
- 提供案件类型选择（三个选项：拖欠工资、劳动合同纠纷、工伤赔偿）。
- 点击"开始咨询"后创建会话。
- 创建成功后使用 `react-router-dom` 的 `navigate` 跳转到 `/chat/:sessionId`。
- 首页不提供案件简要描述输入框，避免与聊天页中的正式案情描述重复。
- UI 要精致、有法律服务产品的专业感。

### 2. 聊天主页（路由 `/chat/:sessionId`）
- 从 URL 中读取 `sessionId`（使用 `useParams`）。
- 页面加载时先请求会话详情和消息历史。
- 页面至少包含：
  - 当前案件类型显示（顶部或侧边信息区）
  - 消息列表区
  - 输入框
  - 发送按钮
- 消息列表区按时间正序显示。
- 用户消息和助手消息视觉上明显区分（不同颜色/方向/气泡样式）。
- 消息列表应自动滚动到底部。

### 3. 发送消息后的前端行为
- 用户点击发送后，前端先做输入校验，禁止发送空字符串。
- 请求成功后，前端立即展示用户消息。
- 同时展示一个"助手正在回复"占位状态（如 typing 动画）。
- 紧接着重新请求消息历史。
- 历史返回后，用真实历史覆盖当前消息列表，移除占位状态。

### 4. 初次进入会话
- 根据 URL 中的 `sessionId` 请求会话详情。
- 若会话不存在（404），展示友好的错误页面（使用 Ant Design `Result` 组件）。
- 请求消息历史，若当前没有消息，展示空状态提示，鼓励用户发送第一条内容。

### 5. 页面刷新行为
- 已通过 URL 承载 `sessionId`，刷新后自动按 URL 参数恢复状态。
- 刷新后重新拉取会话详情和消息历史。

## 任务拆解

### 1. 定义前端类型
- 在 `types/` 中定义：
  - `CaseType`（枚举：`wage_arrears`、`labor_contract`、`work_injury`）
  - `SessionCreateRequest`
  - `SessionResponse`
  - `MessageCreateRequest`
  - `MessageResponse`
  - `ChatHistoryResponse`
- 不在本阶段定义未使用的复杂类型。

### 2. 实现聊天服务层
- 在 `services/chatService.ts` 中封装：
  - `createSession(data: SessionCreateRequest): Promise<SessionResponse>`
  - `getSession(sessionId: string): Promise<SessionResponse>`
  - `sendMessage(sessionId: string, data: MessageCreateRequest): Promise<MessageResponse>`
  - `getMessages(sessionId: string, limit?: number, offset?: number): Promise<ChatHistoryResponse>`
- 统一使用 `request.ts` 中的基础方法。

### 3. 实现首页
- 实现案件类型选择 UI（可使用 Ant Design `Card` 或 `Radio.Group`）。
- 实现"开始咨询"按钮和创建会话流程。
- 成功后跳转到聊天页。
- 失败时显示错误提示。

### 4. 实现聊天页面状态
- 至少维护以下状态：
  - 当前会话信息
  - 当前消息列表
  - 输入框内容
  - 发送中状态
  - 拉历史中状态
  - 错误信息
- 状态粒度以简单可控为主，不在本阶段引入复杂状态库。

### 5. 实现消息发送流程
- 发送前校验输入非空。
- 成功后将用户消息立即加入界面。
- 显示助手回复中的占位状态。
- 随后请求历史消息，拿到真实助手回复并刷新列表。
- 发送失败时恢复输入能力并提示错误。

### 6. 实现历史消息展示
- 拉取历史时展示加载态。
- 无消息时展示空状态。
- 有消息时按升序渲染。
- 对时间显示不做复杂格式化要求，保证可读即可。

### 7. 异常场景处理
- `404 session not found`：使用 `Result` 组件提示会话不存在或已失效。
- `500`：`message.error("系统繁忙，请稍后重试")`。
- 网络失败：`message.error("网络异常，请检查网络连接")`。
- 历史刷新失败：保留当前已显示的用户消息，同时提示同步失败。
- 发送失败：恢复输入能力并提示错误。

## 默认决策
- 默认案件类型直接使用后端已定义值：
  - `wage_arrears`（拖欠工资）
  - `labor_contract`（劳动合同纠纷）
  - `work_injury`（工伤赔偿）
- 默认首页不采集案件简要描述，避免和聊天页输入重复。
- 默认发送消息成功后立即重新拉一次历史，而不是猜测助手回复内容。
- 默认消息列表用服务端返回结果覆盖本地消息列表，避免重复消息和顺序错乱。
- 默认助手回复中的状态由前端本地占位提供，不等待后端专门的流式接口。
- 默认先做单会话页，不在本阶段做会话列表。
- 默认使用 `react-router-dom` 的 `useParams` 读取 `sessionId`，`useNavigate` 做跳转。

## 验收标准
- 用户可从首页成功选择案件类型并创建会话。
- 创建成功后 URL 跳转到 `/chat/:sessionId`。
- 用户发送消息后可看到自己的消息立即出现。
- 随后可看到助手 Mock 回复出现在消息列表中。
- 页面刷新后消息历史不丢失（通过 URL 中的 sessionId 恢复）。
- 页面能正确处理空状态、发送中状态和基础错误状态。
- 整个主链路不依赖手动刷新页面即可走通。
- 整体 UI 看起来专业、精致。

## 风险 / 阻塞
- 发送消息接口只返回用户消息，前端必须额外拉历史，否则看不到助手回复。
- 当前助手回复固定 Mock 文本，后续接入真实 Agent 后可能需要适配更长内容或更慢响应。
- Agent 未完成，无法展示智能分析结果、要素提取等高级功能。
