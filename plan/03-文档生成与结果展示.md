# 03-文档生成与结果展示

## 目标
- 在聊天主链路基础上，完成文档生成入口、结果展示和下载占位交互。
- 用户在完成聊天输入后，能够触发诉状生成，并看到生成结果、文档元信息和导出状态。
- 本阶段以"结果可见、状态明确、限制透明"为优先，不等待完整导出能力上线。

## 当前状态
- 后端已提供文档相关接口：
  - `POST /api/v1/sessions/{session_id}/generate-document`
  - `GET /api/v1/documents/{doc_id}`
  - `GET /api/v1/documents/{doc_id}/export`
- 当前实现特征如下：
  - 生成接口响应模型为 `DocumentResponse`，HTTP 状态码为 `202`。
  - 虽然路由注释写的是"异步生成"，但当前 `backend/services/document.py` 实际是同步写入 Mock 内容和文件。
  - **文档 `content` 字段始终为空**——后端 `mock_generate_document` 只把 Mock 内容写入文件系统，不写入 DB 的 `content` 字段。前端通过任何接口拿到的 `content` 都是空字符串。这不是偶发情况，是后端已知限制。
  - 当前文档状态字段取值为：`pending | generated | exported`
  - 当前导出接口尚未真正下载文件，只返回 `{"message": "Download feature coming soon"}`。
  - 当前模板 ID 沿用案件类型：
    - `wage_arrears`
    - `labor_contract`
    - `work_injury`

## 范围

### 本阶段纳入
- 在聊天页或结果入口中触发文档生成。
- 生成中的按钮禁用和状态提示。
- 生成成功后的结果展示页。
- 文档标题、状态、文件大小、更新时间等信息展示。
- 正文区域的优雅占位提示（因 content 为空）。
- 下载按钮占位和友好提示。

### 本阶段不纳入
- 真正的文件下载实现。
- 模板列表独立选择页。
- 文档编辑器。
- 文档版本管理。
- 多文档列表管理。

## 接口 / 依赖

### 1. 生成文档
请求：

```json
{
  "template_id": "wage_arrears | labor_contract | work_injury",
  "format": "docx | pdf"
}
```

响应关键字段：
- `id`
- `session_id`
- `template_id`
- `title`
- `status`
- `content`（**始终为空字符串，后端已知限制**）
- `file_url`
- `file_size`
- `created_at`
- `updated_at`

### 2. 获取文档详情
请求方式：
- `GET /api/v1/documents/{doc_id}`

响应字段与 `DocumentResponse` 一致。`content` 同样为空。

### 3. 导出文档
请求方式：
- `GET /api/v1/documents/{doc_id}/export`

当前行为：
- 若文档不存在，返回 `404`
- 若文档未生成完成，返回 `400`（detail: `"Document is not ready for export"`）
- 若文档已生成，当前返回 `{"message": "Download feature coming soon"}`，不是真实下载流

## 页面与交互决策

### 1. 生成入口
- 生成入口放在聊天主界面中，默认命名为"生成诉状"。
- 仅当已有有效 `sessionId` 时允许点击。
- 默认模板 ID 直接复用当前会话的案件类型。
- 默认格式先使用 `docx`，不在第一版让用户切换复杂选项。

### 2. 生成中的状态
- 点击生成后立即进入"生成中"状态。
- 按钮禁用，避免重复点击。
- 页面显示清晰的中文提示，例如"正在生成诉状，请稍候"。
- 若请求失败，恢复按钮可点击并展示错误提示。

### 3. 结果展示承载方式
- 生成成功后跳转到结果页 `/result/:docId`。
- 结果页与聊天主区职责分开，避免消息流和文档结果互相干扰。
- 结果页提供"返回聊天"链接，方便用户继续对话。

### 4. 结果页内容（路由 `/result/:docId`）
- 从 URL 读取 `docId`，加载时拉文档详情。
- 结果页至少展示：
  - 文档标题（如"民事起诉状"）
  - 文件大小（若后端有值）
  - 更新时间
  - 生成状态标签
- **正文区域**：由于 `content` 始终为空，默认展示优雅的占位提示，如"文档内容预览功能开发中，请通过下载获取完整文档"。不要展示空白区域。
- 文档不存在时用 `Result` 组件展示错误。

### 5. 文档状态映射
- `pending`：展示为"生成中"
- `generated`：展示为"已生成"
- `exported`：展示为"已导出"
- 其他未知状态：展示为"状态未知"

### 6. 下载按钮策略
- 结果页保留"下载 Word"按钮。
- 当前点击后调用导出接口。
- 如果接口返回占位信息或失败，前端不伪造下载成功，而是明确提示：
  - "下载功能正在完善中，敬请期待"
  - 或展示接口返回的错误信息

## 任务拆解

### 1. 定义文档相关前端类型
- 定义：
  - `DocumentGenerateRequest`
  - `DocumentResponse`
  - `DocumentStatus`（枚举映射，含前端可展示的中文标签）
- 不在本阶段定义未使用的复杂类型。

### 2. 建立文档服务层
- 在 `services/documentService.ts` 中封装：
  - `generateDocument(sessionId: string, data: DocumentGenerateRequest): Promise<DocumentResponse>`
  - `getDocument(docId: string): Promise<DocumentResponse>`
  - `exportDocument(docId: string): Promise<any>`
- 对常见错误码统一转成页面可用的错误信息。

### 3. 建立结果页面状态
- 至少维护以下状态：
  - 当前文档 ID
  - 当前文档详情
  - 生成中状态
  - 获取详情中状态
  - 导出中状态
  - 错误信息

### 4. 实现生成流程
- 从当前会话获取 `sessionId` 和案件类型。
- 生成请求中默认以案件类型作为 `template_id`。
- 成功后记录文档 ID，跳转到结果页 `/result/:docId`。
- 若响应里文档信息已完整，直接渲染；若不完整，再补一次详情请求。

### 5. 实现结果展示
- 展示标题、状态、文件大小和更新时间。
- 正文区域展示占位提示，不展示空白。
- 对过长内容允许滚动，但不在本阶段做复杂富文本渲染。

### 6. 实现导出占位交互
- 按钮点击后调用导出接口。
- 若当前接口只返回占位文本，前端明确提示"下载功能待完善"。
- 不在前端伪造 Blob 下载或猜测文件路径。

### 7. 实现异常场景处理
- `404 document not found`：使用 `Result` 组件提示文档不存在。
- `400 not ready for export`：提示文档尚未可下载。
- `500`：`message.error("生成失败或系统繁忙")`。
- 网络失败：`message.error("网络异常")` 并允许重试。

## 默认决策
- 默认用当前案件类型直接映射 `template_id`。
- 默认生成格式先固定为 `docx`。
- 默认结果页不展示后端 `content` 字段内容（因为始终为空），展示占位提示。
- 默认"生成诉状"操作不做轮询，因为当前后端已同步返回可展示结果。
- 默认下载按钮保留，但严格反映后端当前限制，不做假下载。

## 验收标准
- 用户在已有会话下能成功点击"生成诉状"。
- 请求成功后，跳转到结果页并展示文档信息。
- 页面能正确显示文档状态标签。
- 正文区域展示占位提示，不是空白。
- 点击下载按钮后，前端能给出符合当前后端能力的提示，不产生误导。
- 页面能处理生成失败、文档不存在和导出未就绪等情况。
- 整体 UI 看起来专业、精致。

## 风险 / 阻塞
- 后端 `content` 字段始终为空是已知限制，后续接入真实文档生成后需更新展示逻辑。
- 后端接口虽然返回 `202`，但当前实现实际上是同步生成；若后续改成真实异步，前端需要补轮询或异步状态同步。
- 当前导出接口未实现真实下载，结果页必须把限制写清楚。
- 当前模板获取接口未暴露到路由层，前端暂时不能动态拉模板列表。
