# 01-前端工程初始化

## 目标
- 从零创建 `frontend/` 前端工程，使仓库具备可启动、可扩展、可联调的前端基础骨架。
- 为后续聊天主链路、文档生成和联调计划提供稳定的工程基础，不在本阶段实现业务页面逻辑。

## 当前状态
- 仓库当前不存在 `frontend/` 目录。
- 后端接口已可本地启动并联调，接口前缀为 `/api/v1`。
- 前端默认技术路线已确定为：
  - `Vite`
  - `React`
  - `TypeScript`
  - `Ant Design`
  - `react-router-dom`
- 当前仍未确定真实业务页面布局，但不影响本阶段先搭建工程骨架。

## 前置交付确认
- 无前置依赖，这是第一个阶段。
- 确认 Node >= 18 已安装。

## 范围

### 本阶段纳入
- 初始化 `frontend/` 工程。
- 安装并配置基础依赖（含路由库）。
- 建立基础目录结构。
- 建立应用入口、路由配置、布局组件、基础样式。
- 建立 API 请求层基础封装和服务层入口。
- 建立最小可运行的开发环境和启动方式。

### 本阶段不纳入
- 会话创建逻辑。
- 聊天消息发送与历史展示。
- 文档生成和结果展示。
- 真实业务接口联调细节。
- 测试体系深度建设，仅要求保留后续可接入空间。

## 接口 / 依赖
- 后端联调地址默认本地开发环境，API 前缀固定为 `/api/v1`。
- 前端环境变量至少预留一个基础接口地址：
  - `VITE_API_BASE_URL`（默认值 `http://localhost:8000`）
- 前端当前不直接依赖 Agent 服务，仅依赖后端 HTTP 接口。
- UI 组件默认使用 `Ant Design`，不在本阶段引入额外复杂状态管理库。

## 任务拆解

### 1. 初始化脚手架
- 在仓库根目录创建 `frontend/`。
- 使用 `Vite + React + TypeScript` 初始化项目。
- 保留 Vite 默认开发方式，不在本阶段引入 SSR、Next.js 或自定义构建链。

### 2. 安装基础依赖
- 安装 `antd`。
- 安装 `react-router-dom`（阶段 2 需要路由承载 `sessionId`，在此阶段预装并配置好）。
- 安装请求层依赖：默认使用浏览器原生 `fetch` 封装，不额外引入重量级请求库。
- 保留 TypeScript、Vite、React 的默认必需依赖。

### 3. 建立目录结构
- 建立以下基础目录：
  - `src/pages` — 页面组件
  - `src/components` — 可复用组件
  - `src/services` — API 服务层
  - `src/types` — TypeScript 类型定义
  - `src/layouts` — 布局组件
  - `src/assets` — 静态资源
- 若初始脚手架自带无关 demo 文件，应在初始化阶段清理干净。

### 4. 建立应用骨架
- 应用入口使用统一的 `App` 根组件。
- 在 `App` 中配置 `react-router-dom`，建立最小路由表：
  ```
  /                    → HomePage（占位）
  /chat/:sessionId     → ChatPage（占位）
  /result/:docId       → ResultPage（占位）
  ```
- 创建 `MainLayout` 布局组件，包裹所有路由页面：
  - 布局结构：顶部导航栏 + 内容区
  - 顶部导航栏展示项目名称（"劳动者维权 AI 助手"）
- 各页面创建占位版本，展示页面名称即可。
- 页面默认中文文案。
- 建立基础全局样式，不保留脚手架默认演示样式。
- 整体视觉风格应现代、精致，使用合理的配色和间距。

### 5. 建立 API 服务层基础封装
- 在 `services/request.ts` 中创建统一请求封装：
  - 从 `VITE_API_BASE_URL` 读取后端地址
  - 封装 `get`、`post` 方法
  - 统一设置 `Content-Type: application/json`
  - 非 2xx 响应统一抛出包含 `status` 和 `message` 的自定义错误
  - 网络异常统一捕获
- 创建 `services/chatService.ts` 入口文件（空壳，后续填充）。
- 创建 `services/documentService.ts` 入口文件（空壳，后续填充）。
- 创建 `types/index.ts` 入口文件（空壳，后续填充）。
- 本阶段不必写完所有 API，只需把结构和封装模式搭好。

### 6. 建立环境变量配置
- 创建 `frontend/.env.development`：
  ```
  VITE_API_BASE_URL=http://localhost:8000
  ```
- 确保 `request.ts` 正确读取该环境变量。

### 7. 清理和验证
- 确保 `npm install` 可成功。
- 确保 `npm run dev` 可启动，浏览器打开后不白屏、不报错。
- 确保页面展示基础布局和导航栏。
- 确保路由跳转正常（直接访问 `/`、`/chat/test`、`/result/test` 不白屏）。
- 确保 `npm run build` 可执行。
- 工程不再保留默认 demo 文案和默认示例组件。

## 产出要求
- 产出一个可运行的 `frontend/` 工程。
- 产出统一的前端目录结构。
- 产出可扩展的 `services` / `types` / `pages` / `layouts` 基础层次。
- 产出可用的 `request.ts` 基础请求封装。
- 产出已配置的路由表和布局组件，后续直接填充页面内容即可。

## 验收标准
- 仓库中存在 `frontend/` 目录，且结构清晰。
- 运行 `npm install` 后不报依赖错误。
- 运行 `npm run dev` 后可正常打开页面。
- 页面展示布局和导航栏，不保留默认 demo 内容。
- 各路由页面可正常访问，不白屏。
- `services/request.ts` 存在且封装了基础 fetch 逻辑。
- `npm run build` 可正常完成。

## 风险 / 阻塞
- 如果 Node 版本不满足 Vite 或依赖要求，需要先确认本机 Node 版本。
- 如果 `npm` 镜像或网络异常，可能影响初始化和安装速度。
- 如果后续需要严格对齐团队样式体系，本阶段的页面壳应保持克制，避免过度设计。

## 默认决策
- 默认使用 `Vite + React + TypeScript`。
- 默认使用 `Ant Design` 作为基础 UI 组件库。
- 默认使用 `react-router-dom` 作为路由库。
- 默认不引入额外状态管理库（如 Redux、Zustand）。
- 默认使用原生 `fetch` 封装，不引入 axios。
- 默认 API 地址使用环境变量读取。
- 默认错误提示使用 Ant Design `message.error()`。
