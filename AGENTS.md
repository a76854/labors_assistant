# 项目协作说明

## 项目定位
- 本项目是“劳动者维权 AI 助手”，目标是通过多轮对话收集劳动争议信息，结合法律分析生成可导出的诉状。
- 当前用户主要负责前端，因此默认优先支持前端相关实现、调试与联调工作。

## 当前仓库状态
- 当前仓库已落地后端基础代码，实际存在 `backend/`、`scripts/`、`requirements-backend.txt`、`.env.example` 等文件。
- 当前仓库仍未落地 `frontend/` 和 `agent/` 目录，因此前端工程仍处于待初始化状态。
- `README.md` 仍包含整体规划信息，但涉及后端联调时，应优先以实际代码为准，尤其是：
  - `backend/main.py`
  - `backend/api/routes.py`
  - `backend/api/schema.py`
  - `backend/README.md`
- 开始修改前，必须先以仓库中“实际存在的文件和目录”为准，不把 README 里的规划结构当作已实现代码。

## 当前后端现状
- 后端为 `FastAPI` 项目，应用入口是 `backend/main.py`。
- 接口统一前缀为 `/api/v1`，本地启动后可通过 `/docs` 查看 Swagger 文档。
- 当前已存在的核心接口包括：
  - `POST /api/v1/sessions`
  - `GET /api/v1/sessions/{session_id}`
  - `POST /api/v1/sessions/{session_id}/messages`
  - `GET /api/v1/sessions/{session_id}/messages`
  - `POST /api/v1/sessions/{session_id}/generate-document`
  - `GET /api/v1/documents/{doc_id}`
  - `GET /api/v1/documents/{doc_id}/export`
  - `GET /api/v1/health`
- 当前后端仍有未完成能力，前端联调时必须注意：
  - 消息回复目前是 Mock 文本，不是真实 Agent 输出。
  - 文档导出下载接口尚未真正实现，当前只返回占位信息。
  - 前端如需推进未完成链路，可加临时适配层或 Mock，但必须清楚标记替换点。

## 默认工作范围
- 若用户未特别说明，默认优先处理前端任务，其次再处理与前端直接相关的接口约定、Mock 数据和联调问题。
- 前端技术栈以 README 约定为准：`React 19`、`TypeScript`、`Ant Design 6`。
- 若后续需要初始化前端工程且用户未指定脚手架，默认使用 `Vite + React + TypeScript`，目录放在 `frontend/` 下。

## 前端实现约束
- 优先服务 MVP 链路：多轮对话、信息确认/编辑、诉状预览、文件下载。
- 页面、组件、服务、类型应分层清晰；优先使用如下结构：
  - `frontend/src/pages`
  - `frontend/src/components`
  - `frontend/src/services`
  - `frontend/src/types`
- 所有接口请求统一收敛到 `services` 层，不在页面组件内散落请求逻辑。
- 所有核心数据结构都应显式声明 TypeScript 类型，避免大面积使用 `any`。
- 优先复用 Ant Design 组件，不重复造基础控件。
- 所有面向用户的默认文案使用中文。
- 接口地址、环境差异配置使用环境变量管理，不硬编码到组件中。

## 前端视觉风格约束

本项目追求**接近原生桌面应用的高级质感**，不接受简陋的扁平网页风格。以下规范基于 Ant Design + 原生 CSS 实现，视觉参考来源见 `uiprompts/1.md`。

### 整体基调
- 追求"系统级真实感"——界面应有物理的厚度、光影和层次，而不是纯扁平色块。
- 默认使用**深色模式优先**设计，同时支持浅色模式。所有颜色必须同时定义 light/dark 两套。
- 整体色调沉稳、专业，符合法律服务类产品气质。

### 毛玻璃与材质
- 侧边栏、卡片、弹层等容器使用 `backdrop-filter: blur() saturate()` 实现半透明毛玻璃效果。
- 不同层级使用不同模糊强度：
  - 底层容器（侧栏/布局壳）：轻模糊，低透明度背景。
  - 主内容区：中等模糊，略高不透明度。
  - 弹层/菜单/Popover：强模糊，高不透明度，配合较重阴影。
- 大面积半透明区域可叠加极淡的噪点纹理（opacity ~0.015），防止色带感并模拟材质质感。

### 光影与边框
- 避免粗重的 1px 实线边框。优先使用 `box-shadow` 模拟 0.5px 级别的细线边框：
  - 浅色模式：`box-shadow: 0 0 0 1px rgba(0,0,0,0.05)`
  - 深色模式：`box-shadow: 0 0 0 1px rgba(255,255,255,0.1)`
- 浮动容器（卡片、弹窗、侧栏）**必须**添加顶部内高光，模拟顶部光照效果：
  - `box-shadow: inset 0 1px 0 0 rgba(255,255,255,0.4)`（浅色模式）
  - `box-shadow: inset 0 1px 0 0 rgba(255,255,255,0.06)`（深色模式）
- 阴影使用分层策略（一层紧贴的锐利阴影 + 一层大范围柔和阴影），创造立体感。

### 排版
- 字体栈：`-apple-system, BlinkMacSystemFont, "SF Pro Text", "Inter", "Noto Sans SC", sans-serif`。
- 强制开启抗锯齿：`-webkit-font-smoothing: antialiased`。
- 小字号（< 14px）使用较松的字间距；标题（> 20px）使用紧凑字间距。
- 中文内容使用 `Noto Sans SC` 或系统默认中文字体。

### 组件风格覆盖（Ant Design 定制）
- **按钮**：主按钮使用细微纵向渐变而非纯色填充，配合内部 0.5px 高光边框和轻阴影。
- **输入框**：使用内阴影（`box-shadow: inset 0 1px 2px rgba(0,0,0,0.06)`）代替粗边框；聚焦时使用柔和的光晕环（`box-shadow: 0 0 0 4px rgba(主题色, 0.2)`），不使用默认的 outline。
- **卡片/容器**：使用大圆角（`border-radius: 12px~16px`），半透明背景 + 毛玻璃效果。
- **列表/表格行**：使用极淡交替底色（`rgba(0,0,0,0.02)`），分隔线缩进到文字起始位置。
- **弹窗/抽屉**：从底部或中心弹出，配合弹簧缓动动画；遮罩层使用 `rgba(0,0,0,0.2)`（不要太暗）。
- Ant Design 主题定制通过 `ConfigProvider` 的 `theme` 配置统一注入，不在组件上散写行内样式。

### 动画与微交互
- 所有状态切换使用 CSS `transition`，默认参数：`transition: all 0.2s ease-out`。
- 按钮点击时添加轻微缩放反馈（`transform: scale(0.97)`）。
- 页面切换、列表加载使用渐入效果。
- 不使用生硬的 linear 缓动；优先使用 `ease-out` 或 `cubic-bezier` 模拟弹簧物理。
- 不引入 Framer Motion 等额外动画库，用 CSS transition/animation 实现所有效果。

### 配色参考
- 主题色：推荐使用偏蓝或深青色系（如 `#1677ff` 或自定义），Ant Design 通过 `token.colorPrimary` 统一管理。
- 背景色：
  - 浅色模式：`#f5f5f7`（Apple 灰白）/ 容器 `rgba(255,255,255,0.8)`
  - 深色模式：`#1e1e1e`（底层）/ `#282828`（容器）/ `#323232`（弹层）
- 文字色：浅色模式 `rgba(0,0,0,0.85)`，深色模式 `rgba(255,255,255,0.85)`。
- 不使用纯黑 `#000` 或纯白 `#fff` 作为大面积背景或文字色。

### 实现原则
- 所有视觉定制在 Ant Design `ConfigProvider` + 全局 CSS 变量中统一管理，不散写行内样式。
- 优先用 Ant Design 的 Design Token 系统覆写默认值。
- 自定义样式写在组件同名 `.css` 或 `.module.css` 文件中，不使用 Tailwind CSS。
- 复杂视觉效果（毛玻璃、噪点纹理、分层阴影）可封装为公共 CSS class 放在全局样式中复用。

## 联调与协作边界
- 后端接口已部分落地，前端联调时默认以 `backend/api/routes.py` 和 `backend/api/schema.py` 为准，不以 README 中的示例描述替代真实协议。
- 涉及接口字段、状态码、导出文件格式时，优先查看现有代码和 `/docs`；没有依据时，不自行臆造最终协议。
- 对于后端尚未实现完整逻辑的接口，可在前端加临时 Mock 或兜底展示，但要明确标出“当前为占位联调逻辑”。
- 涉及法律条文、案例筛选、诉状内容正确性时，前端仅负责展示与交互，不擅自修改业务含义。
- 文件导出相关前端逻辑只负责触发下载、展示进度和错误反馈，内容正确性由后端 / 文档系统负责。

## 修改原则
- 先理解现状，再做最小必要改动。
- 命名清晰，抽象克制，删除无用代码，不保留废弃兼容分支。
- 修改前先确认影响范围；如果发现用户已有未提交改动，避免覆盖。
- 未经明确要求，不执行以下操作：
  - 删除文件或批量覆盖文件
  - `git commit` / `git push` / `git reset --hard`
  - 修改系统设置、环境变量或数据库内容
  - 全局安装或卸载依赖

## 验证要求
- 前端改动后，至少执行与本次变更直接相关的验证，如 `npm run lint`、`npm run test`、`npm run build` 或页面手测。
- 若改动涉及后端联调，优先补充接口级验证，例如本地调用 `/api/v1/health`、查看 `/docs`、或对照实际请求 / 响应结构做手测。
- 若仓库仍处于初始化阶段，暂时无法完成验证，需明确说明未验证的原因和缺失前提。
- 输出结果时应包含：改动文件、验证情况、待确认项（如有）。

## 输出约定
- 全程使用中文。
- 回复保持简洁、直接、可执行。
- 引用文件时给出具体路径。

## 文档维护
- 项目根目录的 `FRONTEND_WORKFLOW.md` 用于维护前端任务标准流程和功能实现记录。
- 每次完成一个前端功能、重要页面改动或联调节点后，都要追加一条记录，至少写明目标、改动内容、涉及文件、验证结果和下一步。
- 项目根目录的 `plan/` 目录用于维护规划文件，默认不纳入 Git 版本控制。
- `plan/总计划.md` 是总入口，负责维护当前前端主线目标、阶段推进、分计划索引和风险依赖。
- 后续新增规划默认放在 `plan/` 目录下，按功能模块拆分，不在根目录散落新的规划文档。
- 规划职责与记录职责分离：
  - `plan/` 负责总计划和分计划
  - `FRONTEND_WORKFLOW.md` 负责已完成工作的实现记录和验证结果
- 开始一个新功能前，先更新 `plan/总计划.md` 或对应分计划；功能完成后，再补 `FRONTEND_WORKFLOW.md` 记录。
